# Confirmed-real byte->glyph substitution pairs for the PC Fútbol 6.x / PC
# Apertura DBC character codec (PLAN.md Appendix A, §9 risk #1) — EXPANDED
# via cross-referencing the "foreign reference clubs" stub table.
#
# PROVENANCE (different from confirmed_real_map.txt): every mapping in
# *this* file is derived from decoding all 473 stub records inside the
# real, user-owned `/c/PCF6AR/DBDAT/EQ003003.PKF` (see
# fixtures/PKF_FORMAT.md §3 for the container/record format), using
# `crates/pcf-codec/examples/dump_stub_table.rs` to print each record's
# `short_name` and `stadium_name` fields (lossy-decoded: unmapped bytes
# render as `[XX]`), then identifying the gaps by cross-referencing against
# real, publicly-known football club/stadium names -- both the community's
# `fixtures/pointers/team_pointers.csv` pointer catalog AND, once it became
# clear the stub table's physical order is NOT the same as that catalog's
# fixed pointer order (see note below), plain football-history knowledge of
# the 1998-99 club rosters/stadiums the stub table actually encodes (its
# physical order runs Spain's 1997-98 La Liga table, then Serie A, Premier
# League, Bundesliga, Ligue 1, Primeira Liga, and on through dozens of
# countries in roughly the same country sequence as team_pointers.csv,
# before ending mid-partial-block with special entries like "Jugadores
# Libres"). `confirmed_real_map.txt` (37 pairs, from the "Editor PCF 6.0"
# manual's hex-editing appendix) is a DIFFERENT, independent source and is
# kept separate -- this file is a superset that includes and re-verifies
# all 37 of those pairs against this much bigger corpus (473 records,
# ~940 decoded string fields) with ZERO contradictions found for any of
# them.
#
# METHODOLOGY: for every byte value that appeared as `[XX]` in a lossy
# decode, its glyph was inferred from the one legible, real-world name it
# had to complete (e.g. "At[09]letic Club" can only be "Athletic Club" ->
# 09=h), then CROSS-CHECKED against every other occurrence of that same
# byte anywhere else in the corpus -- across different countries/languages
# where possible, to rule out coincidence. Every pair below with 2+
# citations was confirmed self-consistent across ALL its occurrences (no
# byte was ever found decoding to two different letters). One genuine
# byte-reuse oddity was found: a single Azerbaijani stadium name decodes to
# "Tofi[11] Bakhramov" where the real name is "Tofiq Bakhramov" -- would
# need 0x11='q', but 0x11='p' is independently confirmed by >110 other
# occurrences across a dozen languages (Camp Nou, Sporting, Perugia, Park,
# Apoquindo, Pachencho...). This is almost certainly a data-entry quirk in
# the original 1998 game's own database (p/q mixup when transliterating
# Azeri), NOT a codec ambiguity -- 0x11=p is kept as-is. See
# fixtures/PKF_FORMAT.md's "Character map expansion" section for the full
# writeup.
#
# UPDATE (third pass, EDITOR-PM9798 cross-reference): 8 more pairs added --
# 0x56 (resolving the previously-open San Martín SJ blocker), 0x47, 0x39,
# 0x83, 0xA6, 0xA8, 0xB0, 0xBD -- bringing the total to 90 confirmed pairs.
# See the dedicated section near the end of this file and
# PKF_FORMAT.md's "Character map: EDITOR-PM9798 cross-reference" section.
#
# CONFIDENCE TIERS (all pairs are believed correct; this just reflects how
# much independent evidence backs each one):
#   - No tag: 3+ independent-word citations, zero contradictions.
#   - "2 citations": exactly two independent-word citations.
#   - "1 citation": a single occurrence in the whole corpus. Still a
#     precise, unambiguous real-world name match, but only one data point
#     -- flagged per the task's request to be upfront about single-citation
#     inferences rather than quietly presenting them as equally strong.
#   - "1 citation, same stadium referenced twice": literally one underlying
#     real-world string (e.g. two different clubs sharing a home ground),
#     seen at two file offsets -- more than a single occurrence in the
#     file, but not fully independent evidence.
#
# OPEN QUESTION (deliberately NOT included below): byte 0xD5 appears once,
# in "[2B]ose[D5] Go[0C]es" (Estrela da Amadora's stadium, expected "José
# Gomes"). It sits right where an accented e/é would complete "José", but
# 0x88 is already independently confirmed as é (from the original 37-pair
# manual corpus, re-confirmed here in "Santiago Bernabéu" etc.) -- two
# different bytes both meaning "é" would be an unusual (though not
# impossible) font-table duplication, and one occurrence isn't enough to
# tell whether it's a distinct glyph, a data-entry variant, or noise. Left
# unresolved rather than guessed.
#
# Format matches CharMap::parse: `HH\tChar` (byte hex, tab, glyph; \s means
# space).
